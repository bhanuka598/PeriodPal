# 🤝 Contributing Guidelines – PeriodPal

To maintain code quality, consistency, and proper Git workflow, please follow these rules.

---

## 🔒 Pull Request Policy

🚨 **DO NOT create Pull Requests without repository owner authorisation.**

- No member is allowed to merge code directly into `main` or `dev` branches.
- Pull Requests must be created **only after approval** from the Repository Owner / Team Lead.
- Unauthorised PRs will be rejected.

---

## 🌿 Branching Strategy

All development must be done in feature branches.

Branch naming format:

feature/component-name
bugfix/issue-name

Examples:

feature/user-management  
feature/product-requests  
bugfix/login-error

---

## ✅ Steps Before Creating a Pull Request

1. Complete your feature in your branch
2. Test all endpoints locally
3. Ensure no console errors
4. Ensure code follows project structure
5. Get approval from repo owner
6. Then create the Pull Request

---

## ❌ Strictly Prohibited

- Direct commits to `main`
- Direct commits to `dev`
- Creating PRs without permission
- Pushing `.env` file
- Breaking folder structure

---

## 🧪 Code Quality Rules

- Use meaningful commit messages
- Follow controller → route → model structure
- Proper error handling in APIs
- Follow RESTful conventions
